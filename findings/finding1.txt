Step 3 — Project Comparison
- Architecture:
  - Project 1 is more library‑oriented with explicit data outputs and reusable pipeline components.
  - Project 2 is more script‑oriented with all logic wired in a single main.py.
- Detection approach:
  - Both use YOLOv8; Project 1 includes multiple weights and ONNX options.
- Tracking approach:
  - Project 1 uses BoTSORT (stronger ID persistence).
  - Project 2 uses ByteTrack (lighter, simpler).
- Homography method:
  - Project 1 uses learned keypoints + dynamic homography per frame.
  - Project 2 uses fixed 4‑point mapping, less robust to camera changes.
- Analytics capability:
  - Project 1 outputs structured per‑frame data suitable for downstream analytics.
  - Project 2 has speed/distance and possession overlays but lacks a structured export.
- Code quality:
  - Project 1 is more modular, with clear data contracts in /Users/rupayan/FootballOpenCVProjectSample1/docs/data.md.
  - Project 2 has useful modules but is more demo‑style, with hardcoded paths and transform points.
- Reusability:
  - Project 1 is more reusable for production data pipelines.
  - Project 2 is better as reference for overlays and lightweight estimation.

Recommendation
- Use Project 1’s homography and data schema as the backbone.
- Use Project 2’s camera movement estimator and speed/distance components as analytics modules.

How FastAPI connects
- FastAPI should expose:
  - POST /inference to accept video or frame batches.
  - POST /tracking to return tracks in the canonical schema.
  - POST /calibration for pitch homography diagnostics.
- FastAPI runs the CV pipeline and writes normalized outputs to the database, plus optional JSON export for offline use.

Database recommendation
- Use PostgreSQL with TimescaleDB extension.
- Reason: trajectories are time‑series; TimescaleDB provides compression, fast window queries, and hypertables.

Recommended schema (core tables)

CREATE TABLE matches (
  id UUID PRIMARY KEY,
  competition TEXT,
  season TEXT,
  match_date DATE,
  home_team TEXT,
  away_team TEXT
);

CREATE TABLE videos (
  id UUID PRIMARY KEY,
  match_id UUID REFERENCES matches(id),
  source_path TEXT,
  fps INT,
  width INT,
  height INT
);

CREATE TABLE entities (
  id UUID PRIMARY KEY,
  match_id UUID REFERENCES matches(id),
  entity_type TEXT, -- player, goalkeeper, ball, referee
  team_id INT,
  name TEXT
);

CREATE TABLE tracking_points (
  match_id UUID,
  entity_id UUID REFERENCES entities(id),
  frame_idx INT,
  ts TIMESTAMPTZ,
  x FLOAT,
  y FLOAT,
  x_img FLOAT,
  y_img FLOAT,
  confidence FLOAT,
  PRIMARY KEY (match_id, entity_id, frame_idx)
);

CREATE TABLE events (
  id UUID PRIMARY KEY,
  match_id UUID REFERENCES matches(id),
  ts TIMESTAMPTZ,
  event_type TEXT,
  team_id INT,
  player_entity_id UUID REFERENCES entities(id),
  metadata JSONB
);
